# 08 — 路线 Roadmap

> Pulse 的交付顺序遵循 **B → A → C** 原则：先打数据底子、再串通最小 MVP、最后扩展 UI。本文档定义 MVP 范围、里程碑目标、以及后续版本的优先级队列。

## 一、总体顺序（B → A → C）

| 阶段 | 重点 | 用户价值 | 工程重点 |
|---|---|---|---|
| **B.** 数据底子 | schema + collector | 用户看不到变化 | 决定长期可维护性 |
| **A.** MVP | 三件套可视化 | 能打开看到东西 | 验证"是否每天打开一次" |
| **C.** 扩展 | 多页面 / 多功能 | 深度价值 | 有真实数据后做图表 |

**为什么不是直接做 A？** 如果先做仪表盘，采集会为了快出 UI 偷懒，等仪表盘成型后发现 schema 不够用，就要回头改 —— 改 schema 就意味着重新 migrate 或丢历史数据。对一个"长期采集"的 app 这是致命的。

## 二、MVP（0.1）范围

**目标**：验证"用户是否每天打开一次"。

### 包含功能

| ID | 功能 | 为什么在 MVP |
|---|---|---|
| F-02 | 应用使用时长排行 | 最直观的"看到自己一天做了什么" |
| F-03 | 时段热力图（24h × 7d） | 一眼看出"我什么时候最活跃" |
| F-07 | 指针里程表（戏剧性对比） | **核心上瘾点**：把枯燥数据变有趣 |
| F-05 | 实时仪表盘（菜单栏 + 主窗口） | 让采集状态可见 |
| F-01 | 每日趋势图（基础版） | 连续 7 天数据后开始有意义 |
| F-46 | 敏感时段快捷键 | 隐私底线的最小入口 |
| F-49 | 自检状态页 | 长期跑的采集 app 必备 |

### 包含底层

- 采集：鼠标位移累计 + 前台 App 切换 + 窗口标题哈希 + 按键总数
- 存储：L1 + L2 + L3 聚合表 + 降采样作业
- 引导：Input Monitoring + Accessibility 权限流程 + 隐私承诺勾选
- 发布：Developer ID 签名 + notarization + Sparkle

### **不**包含（明确推迟）

- 鼠标轨迹可视化（F-04 → v1.1）
- 键盘热力图（F-08 → v1.2）
- 周报/月报导出（F-06 → v1.1）
- 所有 C 组扩展采集（电池、相机、剪贴板等）
- 所有衍生指标（晨型/夜型、数字体重）

### MVP 验证指标

| 指标 | 目标 | 测量方式 |
|---|---|---|
| **D7 打开率** | 安装后 7 日内平均 ≥ 1 次/日 | 本地埋点（用户可查看，不上报） |
| **7 日留存** | ≥ 40% | 继续运行采集 |
| **"哇"时刻** | 至少 1 次截图分享社媒 | 社交聆听（手动） |
| **崩溃率** | < 0.1% | Crash log（用户手动反馈） |

如果 MVP 验证不过，先迭代可视化吸引力再扩功能，不是盲目加 B/C 组功能。

## 三、里程碑拆分

### B1 — 数据底子（2 周）

- [ ] Xcode 工程骨架 + SPM 依赖（GRDB, Sparkle）
- [ ] SQLite schema V1：`raw_mouse_moves`, `sec_*`, `min_*`, `hour_*`, `system_events`, `display_snapshots`
- [ ] Migration 框架 + 测试
- [ ] CollectorCore 基础：`EventTapWorker`, `AppWatcher`, `IdleDetector`
- [ ] Aggregator 作业：roll_raw_to_sec / roll_sec_to_min / roll_min_to_hour / purge
- [ ] 多显示器归一化坐标体系（决策见 `04-architecture.md#4.1`）
- [ ] 单元测试 + 24h 模拟跑压测

### B2 — 权限与隐私骨架（1 周）

- [ ] 首次启动引导（欢迎 → 隐私承诺 → Input Monitoring → Accessibility → 完成）
- [ ] 权限自检 + 菜单栏状态图标
- [ ] 一键暂停 + 敏感时段快捷键
- [ ] 偏好设置面板（采集开关、数据删除、加密导出占位）
- [ ] 窗口标题默认哈希化

### A — MVP 串通（2 周）

- [ ] 菜单栏 app 结构（LSUIElement=1 + SMAppService 登录启动）
- [ ] DashboardView：应用排行 + 时段热力 + 里程表 + 今日实时数字
- [ ] 每日趋势图（近 7 天）
- [ ] HealthPanel（自检状态页）
- [ ] 戏剧性对比库（里程 → 地标，可扩展）
- [ ] Info.plist 的权限文案 + usage description
- [ ] Developer ID 签名 + notarize 打包脚本
- [ ] Sparkle 接入 + 首个 appcast.xml

### 冻结期（1 周）

- [ ] 30 人内测（开发者朋友 + Twitter 小范围）
- [ ] 7 天稳定性观察：崩溃、权限掉失、降采样正确性
- [ ] 修关键 bug，不加新功能

## 五、风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| macOS 下一版撤销某项 API | 采集失效 | 每次 macOS beta 跑自动化冒烟；提前申请 entitlement 变更 |
| Apple 私有 API 被拒绝公证 | F-39 可能永远做不成 | v3.0 前评估，不做就删 |
| 用户嫌引导太长 | onboarding 流失 | 把"跳过可选权限"做得非常显眼；关键路径 ≤ 3 屏 |
| 长期数据库膨胀 | 磁盘告急 | 降采样 + 用户可见"清理建议"；监测 DB 大小超过 200MB 弹提醒 |
| CGEventTap 在 Intel Mac 上耗电高 | 电池用户流失 | 节能模式（只记分钟聚合，不记 L0） |
| 定价策略不定 | 无法持续投入 | 到 v1.0 前回到 `07-distribution.md#六` 拍板 |
