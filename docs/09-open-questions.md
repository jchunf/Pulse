## Q-01 · "专注度"指标如何定义？ 🟡 未决

### 背景

用户最初提的定义："应用切换频率低 = 专注、高 = 碎片化、中间 = 多任务。" 直接用这个会有以下问题：

- 看视频/看长文时鼠标键盘都不动、不切应用 → 会被判为"超级深度专注"
- 程序员写代码在 VSCode / Chrome / Terminal / Slack 之间频繁切 → 会被判为"碎片化"，但这其实是正常工作流

### 选项

| # | 方案 | 优点 | 缺点 |
|---|---|---|---|
| A | 保留原定义，简单展示 | 开发最快 | 指标会被用户质疑 |
| B | 按应用类别分组（浏览/开发/视频/沟通） | 更准确 | 需要维护分类规则，有误判 |
| C | 用户可调阈值 + 自定义"深度专注应用白名单" | 尊重用户差异 | UI 变复杂，首次使用门槛高 |
| D | 组合：B 的自动分类作为默认 + C 的白名单作为进阶 | 兼顾 | 最多代码 |

### 建议

采用 **D**。自动分类（B）满足普通用户，白名单（C）满足极客用户。默认分类基于 `UTTypeConformsTo` + 内置的 `bundleId → category` 表（参考 RescueTime 的分类库）。

### 不拍板的后果

F-09 无法实现，或者实现一个会被用户吐槽的版本。

## Q-02 · 是否采集 WiFi SSID？ 🟡 未决

### 背景

F-35（位置画像）基于 WiFi SSID 推断"家/公司/咖啡店"。但开启需位置权限，且 SSID 本身有一定敏感性。

### 选项

| # | 方案 |
|---|---|
| A | 不做 F-35，完全不碰位置 |

| C | 做，但只存 SHA-256 哈希（用户起本地别名） |

### 建议

**B + C 结合**。默认关闭 + 启用即哈希化 + 本地别名。但把它放到 v2.1，不是紧急优先级。

### 不拍板的后果

F-35 卡在设计阶段。

## Q-03 · 通知中断统计要不要做？ 🟡 未决

### 背景

F-38（通知次数）诱人 —— 能量化"你一天被打断几次"。但 macOS 没有公开 API 能干净地拿到通知计数：

- 方案 A：监听 `NSUserNotificationCenter` → 只能拿自己 app 的通知
- 方案 B：读 `~/Library/Group Containers/.../db/db` 的通知库 → 私有实现，macOS 升级可能失效
- 方案 C：通过 `DNServer`（Do Not Disturb 服务）的通知 → 只能拿 DND 开关状态，拿不到具体数量

### 选项

| # | 方案 |
|---|---|
| A | 不做 |
| B | 只做 DND 状态（退化版） |
| C | 读私有 DB（高风险） |

### 建议

**B** 作为 v2.0 的范围（F-37 Focus Mode 联动已经依赖这个），**C 暂不做**。需要时再用"用户手动打卡"方式补充（"我刚被打断了"按钮）。

### 不拍板的后果

F-38 保持在 v3.0 队列，不影响 MVP。

## Q-04 · 年终分身报告能否生成"可分享卡片"？ 🟢 倾向保留

### 背景

F-24（数字分身年度报告）最大价值是"可分享"（像 Spotify Wrapped 刷屏）。但 Pulse 承诺"零上传" —— 生成可发朋友圈的图片卡片是否违背原则？

### 分析

- 在本地生成 PNG/JPEG 图片文件 → **不违背**（数据仍在本地磁盘）
- 把生成的图片**自动**上传到任何服务 → 违背
- 由用户手动在"聚焦 / 访达 / 信息 / 邮件" 等 app 分享 → 不违背（是用户主动行为）

### 建议

**保留，但边界明确**：只生成本地图片文件；用户选择分享时通过系统自带 `NSSharingService` 弹窗（iOS/macOS 标准行为），Pulse 不内嵌任何上传通道。

### 不拍板的后果

F-24 保持模糊，v3.0 做的时候容易变形。

## Q-05 · 定价策略？ 🔴 未决（需在 v1.0 前）

### 背景

见 `07-distribution.md#六`。四种方案各有利弊。

### 建议

暂定 **"一次性买断 + 免费基础版"**：
- 基础版（永久免费）：MVP + v1.1 的功能
- Pro 版（$19–29 买断）：v1.2 之后的所有功能（键盘热力、里程彩蛋、阈值警报、加密导出、Shortcuts）
- 不做订阅

但必须在 v1.0 发布前回到这个问题拍板，否则会出现"免费发了再想收费就晚了"的被动。

### 不拍板的后果

无法商业化；定价过晚会让老用户有心理预期偏差。

## Q-06 · 键码分布默认开还是关？ 🟢 建议关

### 背景

见 `05-privacy.md#4.1`。键码 + 时序有潜在侧信道还原风险。

### 选项

| # | 方案 | 代价 |
|---|---|---|
| A | 默认开启 | F-08 键盘热力图首次启动就能用，但违反"默认保守"原则 |
| B | 默认关闭，引导时单独开关 | F-08 初次打开是空的，用户需二次操作 |
| C | 默认关闭，但提供"启用后 N 天看效果"的轻量引导 | 中间选项 |

### 建议

**B**。这是核心隐私底线，不能为了 UX 妥协。F-08 首次打开展示"尚未收集"占位图，附一个"启用键码收集"按钮即可。

### 不拍板的后果

F-08 的 UX 链路不定，一处隐私原则妥协容易扩散。

## Q-07 · 是否需要 Windows/Linux 版？ 🟢 暂不做

### 背景

Pulse 的很多 macOS 特性（CGEventTap、AXUIElement、SMAppService）没有跨平台等价物。

### 建议

**macOS 专注优先**。如果未来反响好，可以考虑：

- Windows：用 SetWindowsHookEx + RawInput，技术栈全换
- Linux：用 X11 的 `xcb_input`，只适配桌面环境（GNOME / KDE）

但这些需要重写整个 Collector，不走"跨平台一次成"。

### 不拍板的后果

无。仅记录一下。

## Q-08 · 文档本身是否要开源？ 🟡 待定

### 背景

用户要求"所有关键信息存仓库 md 文档"。仓库目前在 GitHub，但可见性（public / private）未明确。

### 选项

| # | 方案 |
|---|---|
| A | 仓库完全 public：代码 + 文档都开源，最大化信任 |
| B | 代码 public、文档 public，但 `09-open-questions.md` 涉及定价的条目隐藏到 private 分支 |
| C | 仓库 private，文档仅内部可见 |

### 建议

**A**（完全 public）最符合 Pulse 的隐私主张 —— "代码可审计"是隐私承诺的兜底。商业敏感信息（定价决策过程、内部讨论）可用另一个 private 项目管理。

### 不拍板的后果

仓库可见性与后续营销策略脱节。

## Q-09 · 个别功能的伦理边界 🟢 建议的原则

### 涉及的潜在争议功能

- F-30 相机/麦克风占用时长 → 可能暴露用户和谁开了会（哪怕只有时长）
- F-32 剪贴板使用频率 → 虽只计数，但计数模式也能推断"是否频繁抄写密码"
- F-39 Space/桌面切换 → 依赖私有 API，有被 Apple 下架风险

### 建议原则

1. **任何功能必须能被单独关闭** — 不打包套餐
2. **默认开启与否以隐私分级为准**（低=开，中/高=关）
3. **私有 API 不进 MVP、不进 v1.x**

## Q-11 · 测试框架：Swift Testing 还是 XCTest？ 🔴 **需在 B1 前拍板**

### 背景

Apple 在 Xcode 16 / Swift 6 推出 Swift Testing（宏驱动、参数化友好），XCTest 仍是旧标准。详见 `10-testing-and-ci.md#七`。

### 选项

| # | 方案 |
|---|---|
| A | 全用 XCTest —— 最稳、文档最多 |
| B | 全用 Swift Testing —— 新项目站队新工具，长期受益 |
| C | 新代码 Swift Testing，XCUITest / 不支持的场景保留 XCTest |

### 建议

**C**。Pulse 定位 macOS 13+，Swift Testing 对最新版本支持好；但 UI 测试（XCUITest）目前只支持 XCTest，必须混用。新写业务逻辑的 PR 一律走 Swift Testing。

### 不拍板的后果

B1 无法开工（测试脚手架是第一步）。

## Q-12 · CI 平台：GitHub Actions 公共 runner vs 自建？ 🟡 **建议 B1 用公共**

### 背景

见 `10-testing-and-ci.md#六`。macOS runner 收费是 Linux 的 10 倍，个人账号免费额度可能不够。

### 选项

| # | 方案 |
|---|---|
| A | GitHub 公共 macOS runner —— 零运维，但量大会付费 |
| B | 自建 Mac mini（M1/M2）挂 actions runner —— 运维成本低、成本可控 |
| C | A 启动，量大后切 B |

### 建议

**C**。B1/A 阶段用公共 runner（每月 CI 运行次数可预估 < 200），发布后若每月成本 > $30/月切自建。

### 不拍板的后果

CI 工作流设计细节会不同（secrets 管理方式、runner 自动扩容策略）。

## Q-13 · 支持的 macOS 版本矩阵？ 🔴 **需在 B1 前拍板**

### 背景

不同 macOS 版本的行为差异大（SMAppService 需 13+、Swift Testing 需 14+、某些 AX API 在 15 有变化）。

### 选项

| # | 方案 | 覆盖用户 | 成本 |
|---|---|---|---|
| A | 仅 macOS 15（最新） | ~40% | 最低 |
| B | macOS 14 + 15 | ~75% | 低 |
| C | macOS 13 + 14 + 15 | ~95% | 中（矩阵 CI 3×） |
| D | macOS 12 + 13 + 14 + 15 | ~99% | 高（SMAppService 要退化到 launchd） |

### 建议

**B**（14 + 15）。13 的用户量正在快速下降；14 是 SMAppService 稳定版；15 是最新。CI 矩阵跑 14 + 15 两个版本。

### 不拍板的后果

最低版本一旦定了，后面降级要写 fallback 代码（launchd plist 等），成本不小。

## Q-14 · 覆盖率硬门槛定多高？ 🟡 **建议 85%**

### 背景

见 `10-testing-and-ci.md#四`。

### 选项

| # | 方案 | 优缺点 |
|---|---|---|
| A | 整体 ≥ 70% | 容易达到，但"刷数字"诱惑大 |
| B | 整体 ≥ 85%，核心模块 ≥ 90% | 业界高水平，有些领域（UI）达到吃力 |
| C | 整体 ≥ 95% | 过度，会写无意义测试 |

### 建议

**B**。对"数据正确性至关重要的模块"（Aggregator、Migration、Collector）≥ 90% 硬门槛，其他 ≥ 80%。视图层靠快照，不算传统覆盖率。

### 不拍板的后果

CI 门槛脚本写死，回头改会影响已有 PR 状态。

## Q-16 · 首次启动是否展示"演示数据"？ 🟡 **建议是**

### 背景

对应 P1"3 分钟内出现第一个哇"（见 `11-ux-principles.md`）。第一天 app 还没积累数据，仪表盘是空的。

### 选项

| # | 方案 |
|---|---|
| A | 空仪表盘 + "请使用 24h 后查看"文字 |
| B | 用本次会话的 30 秒–5 分钟数据即时显示（动态生长） |
| C | 用预置的"示例数据"填满界面，明确标注"示例" |
| D | B + C 组合：先示例，真实数据开始累积后替换 |

### 建议

**D**。这是 Pulse"数据立刻跳动"的 UX 核心。明确标注"示例数据"避免欺骗，真实数据累积到某阈值（例如 20 条事件）后无缝替换。

### 不拍板的后果

P1 目标落空；用户第一天看到空仪表盘，流失风险大增。

## Q-18 · 反馈 / 崩溃报告渠道？ 🟡 **建议 GitHub + 邮件**

### 背景

Pulse 零遥测 = 没有自动 crash 上报。用户遇到问题怎么告诉我们？

### 选项

| # | 方案 |
|---|---|
| A | GitHub Issues + 应用内"复制诊断信息"按钮（预填系统版本、DB 大小、最近错误） |
| B | 专属邮箱 + 同上 |

### 建议

**D**。GitHub 对技术用户友好，邮箱给普通用户。"复制诊断信息"必须严格脱敏（绝不包含窗口标题、应用名等隐私）。

### 不拍板的后果

bug 反馈散在各渠道，难追踪。

## Q-19 · 自检页的"埋点"算不算遥测？ 🟢 **建议属于本地自检**

### 背景

NFR-11 建议"引导完成率"、"首次哇时间"作为 UX 指标。这些数据如果**不上报**、仅本地自检页可见，算不算违反"零遥测"原则？

### 建议

**不算**。只要：
- 数据永不出本地磁盘
- 自检页明确标注"这些数据只在你自己机器上"
- 用户可查看可清除
- 不通过任何网络通道发送

这就是"本地自检"而非"遥测"。文档中用 **"本地可观测性"** 替代"埋点"一词更准确。

### 不拍板的后果

词语上容易被质疑"说好的零遥测呢"。

## Q-20 · CI 自动发 Release 的权限模型？ 🟡 **建议双人审批**

### 背景

tag 触发后 CI 会自动签名、公证、发布 DMG 到 GitHub Releases。这是一个**不可撤销**的对外发布动作。

### 选项

| # | 方案 |
|---|---|
| A | tag 一推，CI 全自动发布 |
| B | tag 推后 CI 构建+公证，但最后一步"发布到 Releases"需要手动点击 |
| C | GitHub Environments 的 `release` environment 开启"Require reviewers"，至少 1 人评审后执行 |

### 建议

**C**。GitHub 原生支持，审批流程丝滑，且评审记录可追溯。

### 不拍板的后果

误 tag 或 CI bug 可能导致有问题的版本流到用户。
